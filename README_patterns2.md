## 1. У чому основна ідея патерну Фасад?

**Фасад (Facade)** — структурний патерн, який надає **спрощений інтерфейс** до складної підсистеми або набору класів. Замість того щоб клієнт взаємодіяв із десятками класів напряму, він звертається до одного об'єкта-фасаду, який бере на себе всю координацію.

```
Клієнт → Фасад → [ ClassA, ClassB, ClassC, ClassD, ... ]
```

```java
// Без фасаду — клієнт змушений знати про всі підсистеми
AudioStream stream = new AudioStream("file.mp3");
Codec codec = CodecFactory.extract(stream);
BitrateReader reader = new BitrateReader();
AudioMixer mixer = new AudioMixer();
reader.read(stream, codec);
mixer.convert(reader);

// З фасадом — один виклик
VideoConverter converter = new VideoConverter();
converter.convert("file.mp3", "mp4");
```

Основна ідея: **приховати складність** і надати зручну точку входу до системи.

## 2. Навіщо ховати подробиці системи від клієнта?

Приховування деталей дає кілька переваг:

- **Зменшення зв'язаності.** Клієнт не залежить від внутрішніх класів підсистеми. Якщо підсистема змінюється — фасад адаптується, а клієнт не змінюється.
- **Спрощення використання.** Клієнту не потрібно розуміти внутрішню логіку — достатньо знати публічний інтерфейс фасаду.
- **Захист від змін.** Внутрішня реалізація може повністю змінитися, але інтерфейс фасаду залишається стабільним.
- **Покращення читабельності коду.** Код клієнта стає коротшим і зрозумілішим.

```java
// Клієнт не знає, що відбувається всередині:
OrderFacade facade = new OrderFacade();
facade.placeOrder(cart, user); // всередині: перевірка складу, оплата, доставка, email
```

## 3. Як патерн Фасад декларує розділення на підсистеми?

Фасад **фізично розділяє** систему на рівні:

- **Зовнішній рівень** — фасад, з яким взаємодіє клієнт.
- **Внутрішній рівень** — підсистеми, які виконують реальну роботу.

```
┌─────────────────────────────────────────────┐
│                   Клієнт                    │
└─────────────────────┬───────────────────────┘
                      │
┌─────────────────────▼───────────────────────┐
│                   Фасад                     │  ← єдина точка входу
└──────┬──────────────┬──────────────┬────────┘
       │              │              │
┌──────▼───┐   ┌──────▼───┐   ┌─────▼────┐
│ Підсист. │   │ Підсист. │   │ Підсист. │
│    A     │   │    B     │   │    C     │
└──────────┘   └──────────┘   └──────────┘
```

Кожна підсистема може існувати незалежно і навіть не знати про інші. Фасад лише **координує** їхню взаємодію. Це відповідає принципу **єдиної відповідальності** — кожен клас робить свою роботу.

## 4. Для якого шару системи варто створювати фасад?

Фасад доцільно створювати **на межі між шарами** програмної системи:

| Шар | Приклад фасаду |
|-----|----------------|
| Бізнес-логіка → Інфраструктура | `RepositoryFacade` |
| Контролер → Сервіси | `ServiceFacade` |
| Зовнішній API → Внутрішня система | `ApiFacade` |
| UI → Бекенд | `BackendFacade` |

Найчастіше фасад створюють для **сервісного шару** — між шаром представлення (UI, контролери) та шаром бізнес-логіки або доступу до даних. Це дозволяє змінювати внутрішні шари без впливу на зовнішні.

> Правило: фасад потрібен там, де взаємодія між шарами стає занадто складною або залежною від деталей реалізації.

## 5. Чим обумовлена наявність двох варіантів патерну Адаптер?

Два варіанти Адаптера — **через об'єкт (Object Adapter)** і **через клас (Class Adapter)** — існують через різні можливості мов програмування.

### Адаптер через об'єкт (є в усіх мовах)
Адаптер **містить** екземпляр адаптованого класу і делегує йому виклики.

```java
class EmailAdapter implements Notifier {
    private LegacyEmailService service; // містить об'єкт

    public void send(String msg) {
        service.sendEmail(msg); // делегує
    }
}
```

### Адаптер через клас (тільки в мовах з множинним спадкуванням)
Адаптер **успадковує** обидва класи одночасно — цільовий інтерфейс і адаптований клас.

```cpp
// C++ — множинне спадкування дозволяє це
class EmailAdapter : public Notifier, public LegacyEmailService {
public:
    void send(string msg) override {
        sendEmail(msg); // метод з LegacyEmailService
    }
};
```

> У Java та C# множинне спадкування класів відсутнє, тому там використовують лише адаптер через об'єкт. У C++ можливі обидва варіанти.

## 6. Як впроваджувати залежності у патерні Dependency Injection?

**Dependency Injection (DI)** — це техніка, при якій об'єкт отримує свої залежності **ззовні**, а не створює їх сам. Існує три основні способи впровадження:

### 1. Через конструктор (Constructor Injection) — рекомендований
```java
class OrderService {
    private final PaymentGateway payment;

    // Залежність передається через конструктор
    public OrderService(PaymentGateway payment) {
        this.payment = payment;
    }
}

// Використання
OrderService service = new OrderService(new StripePayment());
```

### 2. Через сетер (Setter Injection)
```java
class OrderService {
    private PaymentGateway payment;

    public void setPayment(PaymentGateway payment) {
        this.payment = payment;
    }
}
```

### 3. Через поле / інтерфейс (Field / Interface Injection)
```java
class OrderService {
    @Inject // анотація для DI-контейнера (Spring, Dagger тощо)
    private PaymentGateway payment;
}
```

## 7. Як трансформуються залежності при Dependency Injection?

Без DI клас **сам створює** свої залежності — це жорстка прив'язка:

```java
// ❌ До DI — клас залежить від конкретної реалізації
class OrderService {
    private PaymentGateway payment = new StripePayment(); // жорстко зашито
}
```

З DI залежність **інвертується** — клас більше не контролює, що саме він отримує:

```java
// ✅ Після DI — клас залежить від абстракції
class OrderService {
    private final PaymentGateway payment; // залежить від інтерфейсу

    public OrderService(PaymentGateway payment) {
        this.payment = payment; // отримує ззовні
    }
}

// Можна підставити будь-яку реалізацію:
new OrderService(new StripePayment());
new OrderService(new PayPalPayment());
new OrderService(new MockPayment()); // для тестів
```

Трансформація:

```
До DI:   OrderService → створює → StripePayment
Після DI: OrderService ← отримує ← (хтось зовнішній вирішує що підставити)
```

Це реалізує принцип **Inversion of Control (IoC)** — управління залежностями переходить від класу до зовнішнього коду або DI-контейнера.

## 8. Відокремлення абстракції від реалізації в патерні Міст

### Що це означає?

**Міст (Bridge)** розділяє клас на дві незалежні ієрархії:
- **Абстракція** — визначає високорівневий інтерфейс і зберігає посилання на реалізацію.
- **Реалізація** — визначає низькорівневий інтерфейс для конкретних дій.

```
┌──────────────────┐         ┌──────────────────────┐
│   Abstraction    │ ──────► │ <<interface>>         │
│──────────────────│         │   Implementation      │
│ - impl           │         │──────────────────────│
│ + operation()    │         │ + operationImpl()     │
└────────┬─────────┘         └──────────┬───────────┘
         │ extends                      │ implements
┌────────▼─────────┐         ┌──────────▼───────────┐
│ RefinedAbstract. │         │  ConcreteImplA        │
└──────────────────┘         └──────────────────────┘
                              ┌──────────────────────┐
                              │  ConcreteImplB        │
                              └──────────────────────┘
```

### Навіщо це потрібно?

Без патерну Міст при додаванні нових варіантів ієрархія класів росте **експоненційно**:

```
// Без Міст: 2 форми × 3 кольори = 6 класів
RedCircle, BlueCircle, GreenCircle,
RedSquare, BlueSquare, GreenSquare...

// З Мостом: 2 форми + 3 кольори = 5 класів
Circle(color), Square(color) + Red, Blue, Green
```

```java
// Абстракція
abstract class Shape {
    protected Color color; // посилання на реалізацію

    public Shape(Color color) { this.color = color; }
    abstract void draw();
}

// Реалізація
interface Color {
    String fill();
}

class Red implements Color {
    public String fill() { return "red"; }
}

// Розширення незалежне в обох ієрархіях
class Circle extends Shape {
    public void draw() {
        System.out.println("Circle filled with " + color.fill());
    }
}
```

## 9. Чи можливо застосувати декілька ієрархій реалізації у патерні Міст?

**Так**, патерн Міст не обмежує кількість ієрархій реалізації. Абстракція може містити **кілька посилань** на різні інтерфейси реалізації.

```java
abstract class Shape {
    protected Color color;       // перша ієрархія реалізації
    protected Renderer renderer; // друга ієрархія реалізації

    public Shape(Color color, Renderer renderer) {
        this.color = color;
        this.renderer = renderer;
    }

    abstract void draw();
}
```

Це дозволяє комбінувати реалізації незалежно:

```
Форми:    Circle, Square, Triangle
Кольори:  Red, Blue, Green
Рендери:  VectorRenderer, RasterRenderer

// Будь-яка комбінація без вибуху класів:
new Circle(new Red(), new VectorRenderer())
new Square(new Blue(), new RasterRenderer())
```

Головна умова — кожна ієрархія реалізації **незалежна** від інших і підключається через інтерфейс. Таким чином кількість конкретних класів залишається лінійною, а не експоненційною.

## Підсумок

| Патерн | Основна ідея |
|--------|-------------|
| **Фасад** | Спрощений інтерфейс до складної підсистеми |
| **Адаптер** | Перетворює інтерфейс одного класу в інтерфейс, який очікує клієнт |
| **Dependency Injection** | Залежності передаються ззовні, а не створюються всередині |
| **Міст** | Розділяє абстракцію і реалізацію для незалежного розширення |
