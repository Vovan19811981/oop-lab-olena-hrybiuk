## 1. Що виконує фабричний метод у класі?

**Фабричний метод** — це метод, що відповідає за **створення об'єктів**. Він оголошується в базовому класі (або інтерфейсі), але конкретна реалізація — тобто _який саме_ клас буде створено — визначається у підкласах. Метод повертає об'єкт через загальний інтерфейс, не розкриваючи конкретний тип.

Завдяки цьому клас-творець делегує відповідальність за інстанціювання своїм нащадкам.

```java
// Базовий клас з фабричним методом
abstract class Dialog {
    // Фабричний метод
    public abstract Button createButton();

    public void render() {
        Button btn = createButton(); // використовуємо продукт
        btn.onClick();
        btn.render();
    }
}

// Підклас перевизначає фабричний метод
class WindowsDialog extends Dialog {
    @Override
    public Button createButton() {
        return new WindowsButton();
    }
}
```

## 2. Навіщо потрібен фабричний метод?

Фабричний метод вирішує кілька важливих проблем:

### Проблема 1 — Жорстка залежність від конкретних класів
Якщо код напряму викликає `new ConcreteProduct()`, він жорстко прив'язаний до цього класу. Фабричний метод прибирає цю залежність.

```java
// ❌ Погано — жорстка залежність
Button btn = new WindowsButton();

// ✅ Добре — через фабричний метод
Button btn = dialog.createButton();
```

### Проблема 2 — Порушення принципу Open/Closed
Щоб додати новий тип продукту, достатньо створити новий підклас-творець — не змінюючи існуючий код.

### Проблема 3 — Розкидана логіка створення
Якщо умови створення об'єкта складні, їх варто сховати в одному місці, а не дублювати по всій програмі.

## 3. Структура патерну «Фабричний метод»

```
┌─────────────────────────────────────────────────────────────┐
│                        <<interface>>                        │
│                           Product                           │
│─────────────────────────────────────────────────────────────│
│  + doStuff()                                                │
└────────────────────────────┬────────────────────────────────┘
                             │ implements
            ┌────────────────┴────────────────┐
            ▼                                 ▼
  ┌──────────────────┐             ┌──────────────────┐
  │  ConcreteProductA│             │  ConcreteProductB│
  └──────────────────┘             └──────────────────┘

┌─────────────────────────────────────────────────────────────┐
│                     abstract Creator                        │
│─────────────────────────────────────────────────────────────│
│  + someOperation()                                          │
│  + createProduct(): Product   ← фабричний метод             │
└────────────────────────────┬────────────────────────────────┘
                             │ extends
            ┌────────────────┴────────────────┐
            ▼                                 ▼
  ┌──────────────────────┐       ┌──────────────────────┐
  │  ConcreteCreatorA    │       │  ConcreteCreatorB    │
  │──────────────────────│       │──────────────────────│
  │ + createProduct()    │       │ + createProduct()    │
  │   return new         │       │   return new         │
  │   ConcreteProductA() │       │   ConcreteProductB() │
  └──────────────────────┘       └──────────────────────┘
```

Структура складається з **чотирьох ролей**:

| Роль | Опис |
|------|------|
| `Product` | Інтерфейс або абстрактний клас продукту |
| `ConcreteProduct` | Конкретна реалізація продукту |
| `Creator` | Абстрактний клас із оголошенням фабричного методу `createProduct()` |
| `ConcreteCreator` | Підклас, що перевизначає фабричний метод і повертає конкретний продукт |

> Клієнт працює лише з інтерфейсами `Creator` та `Product`, не знаючи про конкретні реалізації.

## 4. Зручність фабрики замість прямого створення об'єктів

Фабрика **ізолює клієнта** від деталей створення. Клієнту не потрібно знати:

- **Який саме клас** буде інстанційовано — може залежати від конфігурації, платформи або середовища.
- **Як саме** налаштовується об'єкт — параметри конструктора, пост-ініціалізація тощо.
- **Скільки** таких об'єктів існує — фабрика може реалізовувати кешування або пул об'єктів.

```java
// Клієнт не знає, який саме продукт отримує
class Application {
    private Dialog dialog;

    void initialize(String os) {
        if (os.equals("Windows")) {
            dialog = new WindowsDialog();
        } else {
            dialog = new WebDialog();
        }
    }

    void main() {
        dialog.render(); // викликає createButton() всередині
    }
}
```

Клієнт викликає метод через інтерфейс і отримує готовий до роботи об'єкт — **без прив'язки до конкретики**.

## 5. Що може приховувати фабрика від клієнта?

Фабрика може приховувати:

- **Конкретні класи** продуктів — клієнт не знає, чи це `WindowsButton`, `MacButton` чи якийсь інший.
- **Логіку вибору** — фабрика сама вирішує, який клас створювати залежно від умов (OS, конфіг, середовище).
- **Складність ініціалізації** — підключення до бази даних, завантаження конфігів, реєстрація в реєстрі тощо.
- **Ієрархію класів** — клієнт взаємодіє лише з інтерфейсом, не бачачи дерева спадкування.

```java
// Клієнт бачить лише це:
Button btn = creator.createButton();
btn.render();

// А фабрика всередині робить:
// - перевіряє OS
// - завантажує тему оформлення
// - реєструє об'єкт в менеджері пам'яті
// - повертає готовий об'єкт
```

## 6. Чому патерн називають «віртуальним конструктором»?

Аналогія пряма: у C++ **конструктори не можуть бути віртуальними** — тобто не можна через базовий клас викликати конструктор підкласу.

```cpp
// У C++ це НЕМОЖЛИВО:
class Creator {
    virtual Creator() {} // ❌ конструктори не бувають віртуальними
};
```

Фабричний метод вирішує цю проблему: це звичайний **віртуальний метод**, який у підкласі створює потрібний об'єкт.

```cpp
// ✅ Фабричний метод як "віртуальний конструктор"
class Creator {
public:
    virtual Product* createProduct() = 0; // віртуальний фабричний метод
};

class ConcreteCreator : public Creator {
public:
    Product* createProduct() override {
        return new ConcreteProduct(); // реальне створення об'єкта
    }
};
```

Таким чином, код, що працює з базовим класом `Creator`, фактично «викликає конструктор» підкласу, не знаючи якого саме — звідси і назва **«віртуальний конструктор»**.

## 7. Відмінності від патерну «Абстрактна фабрика»

| Критерій | Фабричний метод | Абстрактна фабрика |
|----------|-----------------|-------------------|
| **Що створює** | Один тип продукту | Сімейство пов'язаних продуктів |
| **Реалізація** | Метод у класі | Окремий об'єкт-фабрика |
| **Розширення** | Через спадкування (новий підклас) | Через заміну об'єкта-фабрики |
| **Рівень абстракції** | Нижчий | Вищий |

### Фабричний метод
```java
abstract class Creator {
    public abstract Button createButton(); // один метод → один продукт
}
```

### Абстрактна фабрика
```java
interface GUIFactory {
    Button createButton();    // кілька методів →
    Checkbox createCheckbox(); // сімейство пов'язаних продуктів
    TextField createTextField();
}

class WindowsFactory implements GUIFactory {
    public Button createButton()       { return new WindowsButton(); }
    public Checkbox createCheckbox()   { return new WindowsCheckbox(); }
    public TextField createTextField() { return new WindowsTextField(); }
}
```

> **Ключова відмінність:** «Фабричний метод» — це **метод**, «Абстрактна фабрика» — це **об'єкт**. Абстрактна фабрика часто реалізується всередині через кілька фабричних методів.

## Підсумок

```
Factory Method
├── Мета: делегувати створення одного продукту підкласам
├── Механізм: віртуальний метод у базовому класі
├── Розширення: додати новий підклас Creator
└── Коли використовувати:
    ├── невідомо заздалегідь, об'єкти якого класу потрібні
    ├── клас хоче, щоб підкласи визначали тип об'єктів
    └── потрібно централізувати логіку створення
```
